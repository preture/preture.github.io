# 常见传输协议说明

传输协议定义了数据在网络上如何封装、寻址和传输。本文介绍几种最常见的传输层与应用层协议。

## IP 地址与端口号

在网络通信中，**IP 地址**标识一台主机（电脑、服务器、手机），**端口号**标识主机上的具体进程或服务。

```text
192.168.1.100:8080
│───────│ │─│
   IP 地址 端口
```

- **IP 地址**：定位到设备，如 `192.168.1.1`（IPv4）或 `::1`（IPv6）
- **端口号**：0~65535 的整数，知名服务使用固定端口
  - **0~1023**：系统保留端口，需管理员权限绑定
  - **1024~49151**：注册端口，由 IANA 分配或应用自行申请
  - **49152~65535**：动态/私有端口，通常用作客户端临时端口

```bash
# 查看本机 IP 地址
ip addr                    # Linux
ifconfig                   # macOS
ipconfig                   # Windows

# 查看端口占用
lsof -i :8080              # macOS/Linux
netstat -ano | findstr 8080  # Windows
```

## TCP 与 UDP

### TCP（传输控制协议）

面向连接的可靠传输协议，提供数据确认、重传和顺序控制。

```text
客户端                   服务器
  |                       |
  |── SYNC ──────────────→|  建立连接
  |←── SYNC+ACK ─────────|  （三次握手）
  |── ACK ──────────────→|
  |                       |
  |── [数据] ────────────→|  传输数据
  |←── [ACK] ─────────────|
  |── [数据] ────────────→|
  |                       |
  |── FIN ──────────────→|  断开连接
  |←── FIN+ACK ──────────|  （四次挥手）
  |── ACK ──────────────→|
```

**特点：**
- 面向连接（三次握手建立连接）
- 可靠传输（确认重传、去重、顺序保证）
- 流量控制（接收方控制发送速率）
- 拥塞控制（网络拥塞时主动降速）

**适用场景：** Web 浏览（HTTP）、文件传输（FTP）、邮件（SMTP）、远程登录（SSH）

### UDP（用户数据报协议）

无连接的不可靠传输协议，不保证数据到达，但开销极小。

**特点：**
- 无连接（直接发送，无需握手）
- 不可靠（不确认、不重传、不保证顺序）
- 低延迟（没有连接维护和重传机制）
- 支持广播和多播

**适用场景：** 视频直播、在线游戏、DNS 查询、VoIP 语音通话

### TCP vs UDP 对比

| 对比项 | TCP | UDP |
|--------|-----|-----|
| 连接 | 面向连接（三次握手） | 无连接 |
| 可靠性 | 可靠，保证到达 | 不可靠，可能丢包 |
| 有序性 | 保证数据顺序 | 不保证 |
| 速度 | 较慢（有握手和确认开销） | 快（无额外开销） |
| 头部大小 | 20~60 字节 | 8 字节 |
| 端口 | 16 位端口号（0~65535） | 16 位端口号（0~65535） |
| 适用 | 需要完整数据的场景 | 容忍丢包、追求速度的场景 |

## HTTP / HTTPS

HTTP（超文本传输协议）是 Web 通信的基础协议，HTTPS 是加密版本。

### HTTP 请求-响应模型

```http
POST /api/users HTTP/1.1
Host: example.com
Content-Type: application/json
Authorization: Bearer <token>

{"name": "Alice", "email": "alice@example.com"}
```

```http
HTTP/1.1 201 Created
Content-Type: application/json

{"id": 1001, "name": "Alice", "email": "alice@example.com"}
```

### 常见请求方法

| 方法 | 含义 | 幂等 | 安全 |
|------|------|------|------|
| `GET` | 获取资源 | ✅ | ✅ |
| `POST` | 创建资源 | ❌ | ❌ |
| `PUT` | 全量更新 | ✅ | ❌ |
| `PATCH` | 部分更新 | ❌ | ❌ |
| `DELETE` | 删除资源 | ✅ | ❌ |
| `HEAD` | 获取响应头 | ✅ | ✅ |
| `OPTIONS` | 查看支持的请求方法 | ✅ | ✅ |

### 常见状态码

| 状态码 | 含义 | 说明 |
|--------|------|------|
| 200 | OK | 请求成功 |
| 201 | Created | 创建成功（POST） |
| 204 | No Content | 成功但无响应体（DELETE） |
| 301 | Moved Permanently | 永久重定向 |
| 302 | Found | 临时重定向 |
| 400 | Bad Request | 请求参数错误 |
| 401 | Unauthorized | 未认证 |
| 403 | Forbidden | 无权限 |
| 404 | Not Found | 资源不存在 |
| 500 | Internal Server Error | 服务器内部错误 |
| 502 | Bad Gateway | 网关错误 |
| 503 | Service Unavailable | 服务不可用 |

### HTTPS

HTTPS = HTTP + TLS/SSL，对通信内容进行加密，防止窃听和篡改。

```text
HTTP:   http://example.com     port 80    明文传输
HTTPS:  https://example.com    port 443   加密传输
```

TLS 握手流程：

```text
客户端                   服务器
  |                       |
  |── Client Hello ─────→|  支持的 TLS 版本、加密套件
  |←── Server Hello ─────|  选定版本和加密套件 + 证书
  |── 验证证书 ──────────|
  |── 生成密钥 ──────────|
  |←── 确认 ────────────|
  |── 加密通信 ──────────→|
```

## WebSocket

WebSocket 提供浏览器和服务器间的全双工持久连接，由 HTTP 升级而来。

**默认端口：** `ws://` 使用 80 端口，`wss://` 使用 443 端口。

```text
HTTP 请求升级：
GET /ws HTTP/1.1
Connection: Upgrade
Upgrade: websocket
```

```text
建立后双向通信：
    客户端 ←→ 服务器  （全双工，无需反复握手）
```

**特点：**
- 基于 TCP，双向通信
- 一次握手，持久连接
- 头部开销小（仅 2 字节）
- 支持文本和二进制帧

**适用场景：** 实时聊天、协同编辑、股票行情推送、在线游戏

## gRPC

gRPC 是 Google 开发的高性能 RPC 框架，基于 HTTP/2 和 Protocol Buffers。

**默认端口：** 9090（服务端），但可以配置为任意端口。

**特点：**
- 基于 HTTP/2（多路复用、流控、头部压缩）
- 默认使用 Protobuf 序列化
- 支持四种调用模式：
  - 一元 RPC（客户端→服务器，服务器→响应）
  - 服务端流式 RPC（服务器持续推送）
  - 客户端流式 RPC（客户端持续发送）
  - 双向流式 RPC（双方同时发送）

```protobuf
service UserService {
  rpc GetUser (UserRequest) returns (User);
  rpc ListUsers (Empty) returns (stream User);       # 服务端流
  rpc UploadAvatar (stream Chunk) returns (Status);  # 客户端流
  rpc Chat (stream Message) returns (stream Reply);  # 双向流
}
```

**适用场景：** 微服务间通信、移动端与服务器通信、多语言异构系统

## MQTT

MQTT（消息队列遥测传输）是轻量级的发布/订阅协议，专为物联网（IoT）设计。

**默认端口：** 1883（未加密），8883（TLS 加密）。

```text
发布者 ──→ Broker ──→ 订阅者 A
               │
               └──→ 订阅者 B
```

**特点：**
- 发布/订阅模式，解耦发送方和接收方
- 三种 QoS（服务质量）级别：
  - QoS 0：最多一次（可能丢消息）
  - QoS 1：至少一次（可能重复）
  - QoS 2：恰好一次（最可靠）
- 头部最小仅 2 字节
- 支持遗嘱消息（Last Will）和保留消息

**适用场景：** 传感器数据采集、智能家居、移动推送通知
